## 1. What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform originally developed by **Google** and now maintained by the **CNCF** (Cloud Native Computing Foundation). It automates the deployment, scaling, and management of containerized applications.

Google ran billions of containers internally using a system called **Borg**. In 2014, they open-sourced a rewritten version of it and called it **Kubernetes** (Greek for "helmsman" ŌĆö the person who steers a ship).

---

## 2. Why Learn Kubernetes?

- **Containers alone aren't enough** ŌĆö Docker can run containers, but who restarts them when they crash? Who distributes them across servers? Who scales them up during traffic spikes? That's Kubernetes.
- **Industry standard** ŌĆö Almost every company running microservices in production uses K8s (AWS EKS, Azure AKS, Google GKE).
- **Self-healing** ŌĆö If a pod dies, K8s automatically brings it back.
- **Auto-scaling** ŌĆö Scale from 1 pod to 100 pods based on CPU/memory usage.
- **Rolling updates & rollbacks** ŌĆö Update your app with zero downtime, and roll back if something breaks.

---

## 3. Monolithic vs Microservices

### Monolithic Architecture
- The entire application is **one single unit** ŌĆö frontend, backend, database logic, all bundled together.
- If you want to update the login page, you have to **redeploy the entire app**.
- If one part crashes, the **whole app goes down**.
- Example: A single `.war` or `.jar` file running everything.

### Microservices Architecture
- The application is **split into small independent services** ŌĆö login service, payment service, search service, etc.
- Each service runs in its **own container** and can be updated, scaled, or restarted independently.
- If the payment service crashes, the search service **keeps working**.
- Kubernetes is built to manage this kind of architecture ŌĆö it handles hundreds of small containers across multiple servers.

## 4. Kubernetes Architecture

K8s follows a **Control Plane + Worker Nodes** model. Think of it like a shipping port ŌĆö the Control Plane is the office (the bosses), and the Worker Nodes are the ships where the actual work happens.

### Control Plane Components (The Brain)

| Component | What it does |
| :--- | :--- |
| **kube-apiserver** | The front door / receptionist. Every request (from users, nodes, internal components) goes through the API server. It validates and processes all REST operations. |
| **etcd** | The filing cabinet. A distributed key-value store that holds the entire cluster's state and configuration. It's the single source of truth. |
| **kube-scheduler** | The logistics planner. Watches for newly created pods with no assigned node and picks the best node for them based on resource requirements, taints, and affinity rules. |
| **kube-controller-manager** | The supervisor. Runs background controllers like the Node Controller (detects if a node goes down), ReplicaSet Controller (maintains desired pod count), and Job Controller. |
| **cloud-controller-manager** | The liaison to the cloud. Talks to your cloud provider's API (AWS, Azure, GCP) to manage load balancers, storage volumes, and routes. Only present in cloud-managed clusters. |

### Worker Node Components (The Ships)

| Component | What it does |
| :--- | :--- |
| **kubelet** | The ship captain. An agent running on every node that receives instructions from the API server and ensures containers are running healthy inside their pods. It also performs health checks (liveness & readiness probes). |
| **kube-proxy** | The traffic controller. Maintains network rules on each node so that traffic can reach the correct pods. Handles Service-to-Pod routing. |
| **Container Runtime** | The engine. The software that actually pulls images and runs containers. Examples: `containerd`, `CRI-O`, Docker (deprecated as runtime in newer K8s). |

### The Vital Extras

| Component | What it does |
| :--- | :--- |
| **kubectl** | The remote control. A command-line tool you use from your laptop to send commands to the API server. |
| **CNI (Container Network Interface)** | The plumbing system. A plugin that assigns IP addresses to pods and enables pod-to-pod communication across nodes. Examples: `kindnet`, `flannel`, `calico`, `weave`. |

### How it all works together
1. You run `kubectl apply -f deployment.yml` ŌåÆ request goes to **kube-apiserver**.
2. API server validates it and stores the desired state in **etcd**.
3. **kube-scheduler** sees unscheduled pods and assigns them to worker nodes.
4. **kubelet** on the assigned node pulls the container image and starts the pod.
5. **kube-proxy** sets up networking so the pod is reachable.
6. **kube-controller-manager** continuously watches ŌĆö if a pod dies, it tells the scheduler to create a new one.

## 5. Cluster Setup Tools

### KIND (Kubernetes in Docker)
- Runs K8s nodes as **Docker containers** on your machine ŌĆö lightweight and fast.
- Best for **multi-node clusters on a single machine** (e.g., 1 control-plane + 3 workers).
- You define the cluster in a `kind-config.yml` file specifying node roles, images, and port mappings.
- Uses `containerd` as the container runtime inside each node container.
- Great for learning and CI/CD testing, not for production.

### Minikube
- Creates a **single-node** K8s cluster, usually inside a VM or Docker container.
- Heavier than Kind ŌĆö needs more RAM and can struggle on small EC2 instances (4GB RAM).
- Use `--driver=docker` on cloud instances since they don't support nested virtualization (`vmx`/`svm` = 0).
- Has built-in addons like Dashboard, Metrics Server, etc.

### Kubeadm
- The **official tool** to set up production-grade K8s clusters on real VMs or bare metal.
- You manually set up the control plane on one node and join worker nodes using a token.
- Requires you to install the container runtime, kubelet, and kubeadm on each machine yourself.
- Most complex setup but closest to how real production clusters work.

## 6. Namespaces

Namespaces are a way to **divide a single cluster into virtual sub-clusters**. Think of them as separate "offices" inside one building ŌĆö each team gets their own space and their resources don't mix.

### Why use them?
- **Organization** ŌĆö Separate environments like `dev`, `staging`, `prod` inside one cluster.
- **Avoid name collisions** ŌĆö You can have a pod named `web-server` in both `team-a` and `team-b` namespaces without conflict.
- **Resource limits** ŌĆö You can set quotas per namespace (e.g., marketing team gets max 2GB RAM).

### Default namespaces K8s creates automatically:
| Namespace | Purpose |
| :--- | :--- |
| `default` | Where your resources go if you don't specify a namespace. |
| `kube-system` | Internal K8s components live here (API server, scheduler, etcd, kube-proxy, CNI). Don't touch! |
| `kube-public` | Publicly readable data, rarely used directly. |
| `kube-node-lease` | Holds lease objects for node heartbeats ŌĆö helps detect node failures faster. |

### Key behavior:
- If you run `kubectl get pods` without `-n`, it only shows the `default` namespace.
- Use `kubectl get pods -A` to see everything across all namespaces.
- When you delete a namespace, **everything inside it gets deleted** (pods, deployments, services ŌĆö all gone).

## 7. Pods

A Pod is the **smallest deployable unit** in Kubernetes. It's a wrapper around one or more containers that share the same network and storage.

### Key points:
- A pod usually runs **one container** (e.g., one nginx container). Multi-container pods are for special patterns like sidecars.
- Every pod gets its **own IP address** inside the cluster.
- Pods are **ephemeral** ŌĆö they are meant to be created and destroyed. If a pod dies, K8s doesn't "fix" it, it creates a brand new one.
- You should almost never create pods directly. Use a **Deployment** instead, which manages pods for you.

### Pod YAML structure:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  namespace: nginx
spec:                    # NOT "specs" ŌĆö this is a common typo!
  containers:
  - name: nginx-container
    image: nginx:latest
    ports:
    - containerPort: 80
```

### Common gotcha you hit:
- Writing `specs:` instead of `spec:` ŌåÆ gives error: `unknown field "specs"`.

## 8. Deployments

A Deployment is the **most commonly used workload resource** in K8s. It manages a set of identical pods through a ReplicaSet and handles updates and rollbacks.

### What it does:
- Tells K8s "I want 3 copies of this nginx pod running at all times."
- If a pod crashes, the Deployment (via its ReplicaSet) **automatically creates a new one**.
- Handles **rolling updates** ŌĆö when you change the image version, it replaces pods one-by-one with zero downtime.
- You can **scale up/down** anytime by changing the replica count.

### Deployment YAML structure:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  namespace: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
```

### Common gotcha you hit:
- Adding `name:` directly under `template:` ŌåÆ error: `unknown field "spec.template.name"`. The `name` goes inside `template.metadata`, not at the template level.

## 9. Labels and Selectors

### Labels
Labels are **key-value pairs** attached to any K8s resource (pods, nodes, services, deployments). They are used to organize and identify resources.

Example:
```yaml
metadata:
  labels:
    app: nginx
    env: production
    team: backend
```

### Selectors
Selectors are **filters** that use labels to find and connect resources. This is how a Deployment knows which pods belong to it.

In a Deployment YAML:
```yaml
selector:
  matchLabels:
    app: nginx    # "Find all pods that have the label app=nginx"
```

### How they work together:
1. The Deployment's `selector.matchLabels` says: "I own pods with label `app: nginx`."
2. The pod template's `metadata.labels` says: "I am labeled `app: nginx`."
3. If these **don't match**, the Deployment won't manage those pods and K8s will throw an error.

## 10. Rolling Updates

A Rolling Update is how Kubernetes **upgrades your app without downtime**. Instead of killing all old pods and starting new ones (which causes downtime), it does it gradually.

### How it works:
1. You run: `kubectl set image deployment/nginx-deployment nginx=nginx:1.26.3`
2. K8s creates a **new ReplicaSet** with the new image version.
3. It starts new pods in the new ReplicaSet, one by one.
4. As each new pod becomes `Ready`, it kills one old pod from the old ReplicaSet.
5. This continues until all pods are running the new version.

### What you saw in your terminal:
- Old pods showed `Terminating` status while new pods showed `ContainerCreating` ŌåÆ `Running`.
- At no point were all pods down ŌĆö some old ones were still serving traffic while new ones came up.

### Rollback:
If the new version is broken, you can roll back instantly:
```
kubectl rollout undo deployment/nginx-deployment -n nginx
```

## 11. ReplicaSet

A ReplicaSet's only job is to **maintain a stable set of replica pods** running at any given time.

### Key points:
- It's the "basic guard" ŌĆö it just counts pods. "I need 3 pods. I see 2. Let me create 1 more."
- If you delete a pod, the ReplicaSet immediately creates a replacement.
- **You almost never create a ReplicaSet directly.** When you create a Deployment, it automatically creates a ReplicaSet behind the scenes.
- ReplicaSets can't handle rolling updates on their own ŌĆö that's why Deployments exist on top of them.

### The chain: **Deployment ŌåÆ ReplicaSet ŌåÆ Pods**

## 12. StatefulSet

A StatefulSet is for applications that need **stable identity and persistent storage** ŌĆö typically databases.

### How it's different from Deployment:
- Each pod gets a **fixed, predictable name**: `db-0`, `db-1`, `db-2` (not random like `nginx-abc12`).
- Each pod gets its **own persistent volume** (storage). If `db-0` dies and comes back, it reconnects to the same storage.
- Pods are started **in order** (0 ŌåÆ 1 ŌåÆ 2) and deleted in **reverse order** (2 ŌåÆ 1 ŌåÆ 0).
- Used for: MySQL, PostgreSQL, MongoDB, Redis, Kafka, Elasticsearch.

---

## 13. DaemonSet

A DaemonSet ensures that **one copy of a pod runs on every node** (or selected nodes) in the cluster.

### Key points:
- **No `replicas` field** ŌĆö the number of pods = number of nodes. You don't set it manually.
- When a new node joins the cluster, the DaemonSet automatically puts a pod on it.
- When a node is removed, that pod is garbage collected.
- Used for system-level tasks that need to run everywhere.

### Real examples already running in your cluster:
- `kube-proxy` ŌĆö networking rules on every node.
- `kindnet` ŌĆö CNI plugin managing pod networking on every node.

### Other common uses:
- Log collectors (Fluentd, Filebeat) ŌĆö gather logs from every node.
- Monitoring agents (Prometheus node-exporter) ŌĆö collect metrics from every node.

## 14. Jobs

A Job creates one or more pods and ensures they **run to completion** successfully. Unlike Deployments (which keep pods running forever), Jobs are for **one-time tasks**.

### Key points:
- The pod runs, does its work, and exits. K8s marks the Job as `Complete`.
- If the pod fails, the Job **retries** it (configurable with `backoffLimit`).
- Use cases: database migrations, batch processing, sending bulk emails, running a backup script.

### Single vs Batch (Parallel) Execution:

By default a Job runs **1 pod, 1 time**. But you can configure it to run in batches:

| Field | What it does | Example |
| :--- | :--- | :--- |
| `completions` | Total number of times the task needs to finish successfully. | `completions: 10` ŌåÆ run 10 tasks total |
| `parallelism` | How many pods run at the same time. | `parallelism: 3` ŌåÆ run 3 pods simultaneously |

Combine both: `completions: 10, parallelism: 3` ŌåÆ K8s starts 3 pods at a time, as one finishes it starts another, keeps 3 running until all 10 are done.

### Types:
- **Job** ŌĆö Run once (or in batches) and finish. Example: "Run this database migration script."
- **CronJob** ŌĆö Run on a schedule (like Linux cron). Example: "Take a backup every night at 2 AM."

### Job YAML ŌĆö Single run:
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  template:
    spec:
      containers:
      - name: migrate
        image: my-migration-image:latest
      restartPolicy: Never
  backoffLimit: 3          # Retry up to 3 times if it fails
```

### Job YAML ŌĆö Batch / Parallel run:
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: batch-process
spec:
  completions: 10          # Total 10 tasks need to finish
  parallelism: 3           # Run 3 pods at the same time
  template:
    spec:
      containers:
      - name: worker
        image: my-batch-image:latest
      restartPolicy: Never
```

### CronJob YAML example:
```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 2 * * *"   # Every day at 2:00 AM
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: my-backup-image:latest
          restartPolicy: Never
```

## 15. CronJob (Deep Dive)

A CronJob is a **scheduled Job** ŌĆö like setting an alarm clock that runs a task automatically at a fixed interval.

### CronJob Schedule Format:
The schedule has **5 slots** ŌĆö think of it like 5 boxes:

```
ŌöīŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆ Minute (0-59)
Ōöé ŌöīŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆ Hour (0-23)
Ōöé Ōöé ŌöīŌöĆŌöĆŌöĆŌöĆŌöĆ Day of Month (1-31)
Ōöé Ōöé Ōöé ŌöīŌöĆŌöĆŌöĆ Month (1-12)
Ōöé Ōöé Ōöé Ōöé ŌöīŌöĆ Day of Week (0-6, Sun=0)
Ōöé Ōöé Ōöé Ōöé Ōöé
*/2 * * * *
```

| Example | Meaning |
| :--- | :--- |
| `*/2 * * * *` | Every 2 minutes |
| `0 * * * *` | Every hour (at minute 0) |
| `0 0 * * *` | Every day at midnight |
| `0 9 * * 1` | Every Monday at 9 AM |
| `0 0 1 * *` | 1st of every month at midnight |

### CronJob YAML structure (corrected):
```yaml
kind: CronJob
apiVersion: batch/v1
metadata:
  name: minute-backup
  namespace: nginx
spec:
  schedule: "*/2 * * * *"
  jobTemplate:
    spec:
      template:
        metadata:
          name: minute-backup
          labels:                    # labels go INSIDE metadata
            app: minute-backup
        spec:
          containers:
          - name: backup-container
            image: busybox:latest
            command:
            - sh
            - -c
            - >
              echo "Backup started";
              mkdir -p /backups &&
              mkdir -p /demo-data &&
              cp -r /demo-data /backups &&
              echo "Backup finished";
            volumeMounts:
              - name: data-volume
                mountPath: /demo-data
              - name: backup-volume
                mountPath: /backups
          restartPolicy: OnFailure     # pod-level, NOT inside containers
          volumes:                     # pod-level, NOT inside containers
            - name: data-volume
              hostPath:
                path: /demo-data
                type: DirectoryOrCreate
            - name: backup-volume
              hostPath:
                path: /backups
                type: DirectoryOrCreate
```

### Common YAML mistakes to avoid:
- `labels` must be **inside** `metadata`, not at the same level.
- `restartPolicy` is a **pod-level** setting, not inside `containers`.
- `volumes` is a **pod-level** setting, not inside `containers`. Volumes are shared across all containers in a pod.

### How the CronJob chain works:
```
CronJob (the alarm clock)
  ŌåÆ creates a Job (every 2 minutes)
    ŌåÆ Job creates a Pod (the actual worker)
      ŌåÆ Pod runs the command and exits
```

## 16. Volumes

Volumes are how you give **storage** to your containers. Without volumes, all data inside a container is **lost** when the container dies.

### The 3-step volume process:
1. **`volumes`** ŌĆö Define the storage source (like buying a USB drive).
2. **`volumeMounts`** ŌĆö Attach it to a path inside the container (like plugging the USB in).
3. **`command`** ŌĆö Use the mounted path in your app (like opening files on the USB).

### hostPath Volume:
Maps a **folder on the worker node** directly into the container.

```
ŌöīŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöÉ
Ōöé  Worker Node (EC2 / KIND container)     Ōöé
Ōöé                                         Ōöé
Ōöé  /demo-data  ŌåÉ real folder on host     Ōöé
Ōöé       ŌåĢ (connected via hostPath)        Ōöé
Ōöé  ŌöīŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöÉ    Ōöé
Ōöé  Ōöé  Container                      Ōöé    Ōöé
Ōöé  Ōöé  /demo-data ŌåÉ mount point      Ōöé    Ōöé
Ōöé  Ōöé  (reads/writes to host folder)  Ōöé    Ōöé
Ōöé  ŌööŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöś    Ōöé
ŌööŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöś
```

### Key point ŌĆö volumes are created on the WORKER NODE, not the master:
- `kubectl apply` runs on the master node (control plane).
- But the pod runs on a **worker node** ŌĆö that's where `hostPath` folders get created.
- In KIND, worker nodes are Docker containers. Use `docker exec -it <worker-name> ls /path` to see the folders.

### `DirectoryOrCreate` type:
- If the folder exists on the host ŌåÆ use it.
- If it doesn't exist ŌåÆ create it automatically when the pod starts.

## 17. Persistent Volumes (PV) and Persistent Volume Claims (PVC)

### PersistentVolume (PV):
A PV is a **piece of storage** pre-provisioned by an admin. Think of it as a **parking spot** in a parking lot.

```yaml
kind: PersistentVolume
apiVersion: v1
metadata:
  name: local-pv
  labels:
    app: local
spec:
  capacity:
    storage: 1Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: local-storage
  hostPath:
    path: /mnt/data
```

### Important: `storage: 1Gi` with hostPath is NOT a real limit!
- With `hostPath`, the `1Gi` is just a **label/promise** ŌĆö it doesn't enforce any actual disk limit.
- The container can use the **entire host disk** (e.g., 29G on your EC2).
- It's like putting tape on the floor and writing "1Gi" ŌĆö there's no real wall.
- For real storage limits, use AWS EBS (`storageClassName: gp2`) which creates an actual separate disk.

### Access Modes:
| Mode | Meaning |
| :--- | :--- |
| `ReadWriteOnce` (RWO) | One node can read/write at a time |
| `ReadOnlyMany` (ROX) | Many nodes can read, none can write |
| `ReadWriteMany` (RWX) | Many nodes can read/write (needs NFS or EFS) |

### Reclaim Policies:
| Policy | What happens when PVC is deleted |
| :--- | :--- |
| `Retain` | PV keeps the data, admin must manually clean up |
| `Delete` | PV and its data are automatically deleted |
| `Recycle` | Data is wiped (`rm -rf`), PV becomes available again (deprecated) |
